# v0.5.0 Roadmap — "Shareable Scenes"

**Status**: ✅ All phases complete on `feature/v0.5.0-phase7-share-codes`. Awaiting review before tagging v0.5.0.
**Created**: May 10, 2026
**Last updated**: May 11, 2026 (7d + 7e + 7f green; 2317 tests; share codes round-trip end-to-end)
**Focus**: Reproducible scenes — every pattern becomes a seeded function so `splash share` / `splash play <code>` can reconstruct the exact same animation byte-for-byte across machines.

---

## 🎯 Vision

Make ascii-splash **scenes shareable**. Today every pattern silently calls `Math.random()` — restart the same pattern and you get a different rain, a different starfield, a different fish swimming past. v0.5 makes that randomness deterministic and threadable through a u32 seed. The first user-visible feature is share codes; the foundation also unlocks asciinema-style replay, time-of-day automation, and any future feature that needs reproducibility.

---

## ✅ In Scope

- ✅ **Injected `Random` abstraction** — replaces every `Math.random()` site in `src/patterns/` with a constructor-injected PRNG
- ✅ **`splash share`** — print a base32 code for the current `{patternId, presetId, themeId, seed, configHash}`
- ✅ **`splash play <code>`** — decode + run; byte-for-byte replay
- ✅ **In-app `S` key** — copy current state's share code to clipboard
- ✅ **Format versioning** — version-prefix byte so future formats reject gracefully

## ❌ Out of Scope (deferred to v0.6+)

- ❌ asciinema `.cast` export (Phase 8 of the v0.4 roadmap) — separate concern, can ride on top of share codes once they land
- ❌ GIF export (Phase 9 of the v0.4 roadmap)
- ❌ Native graphics protocols (Phase 5 of the v0.4 roadmap — Kitty / iTerm2 / Sixel pass-through)
- ❌ Color-mask sprites (Phase 6 of the v0.4 roadmap)
- ❌ Anything that requires cross-process determinism beyond a single replay (multi-user sync, etc.)

## 🛠️ Design Decisions (locked)

1. **Random is a required constructor arg.** Pattern constructors are `(theme, random, config?)` — theme first, random second, optional config last. Not optional, no Math.random fallback.
2. **One `Random` per pattern instance.** Each pattern's sequence is independent, so a share code can reseed a single pattern without disturbing the rest of the app.
3. **`Mulberry32`** (Tommy Ettinger, public domain) is the default impl. ~10 LOC core, u32 state, period 2^32, statistical quality sufficient for visual use. Implementations are pluggable via the `Random` interface.
4. **`randomSeed()`** bootstrap helper uses `Math.random()` — the one allowed call, isolated to `src/utils/random.ts`, to pick a fresh u32 when the user hasn't supplied a share code.
5. **CommandExecutor stays on `Math.random()`.** Surprise-me commands are intentionally non-deterministic; they're UX, not scene state.
6. **Reference vectors locked.** `tests/unit/utils/random.test.ts` pins the first 5 outputs of `seed=1` so any future PRNG swap has to ship a migration.

**Commit**: `c4e4ae7`
**Shipped**:

## 📝 Migration Recipe (for in-batch work)

For each pattern file:

1. Add `import { Random } from '../utils/random.js';`
2. Add `private random: Random;` field
3. Constructor signature `(theme: Theme, random: Random, config?: ...)`
4. In constructor body `this.random = random;`
5. Replace each `Math.random()` site:
   - `Math.random()` → `this.random.next()`
   - `Math.random() * N` → `this.random.next() * N`
   - `min + Math.random() * (max - min)` → `this.random.range(min, max)`
   - `Math.floor(Math.random() * arr.length)` indexing → `this.random.choice(arr)` (throws on empty; guard inline if a test relies on graceful-empty behavior — see `RainPattern.pickChar()` for the canonical pattern)
   - `Math.floor(Math.random() * N) + offset` integer → `this.random.int(offset, offset + N - 1)` (note: `int` is **inclusive** on both ends, unlike `Math.floor(Math.random() * N)` which is `[0, N-1]`)
   - `Math.random() < p` → `this.random.bool(p)`
6. Update `src/main.ts` construction site: pass `new Mulberry32(randomSeed())` as the 2nd arg
7. Update every test file that constructs the pattern to pass `new Mulberry32(42)` (fixed seed for determinism) + add `import { Mulberry32 } from '../../../src/utils/random.js';`

After each batch: `npm run typecheck && npm test`. Both clean before commit. Commit message convention:

```
feat(v0.5.0): phase 7c batch N — <pattern list>
```

Commitlint enforces lowercase subject, header ≤72 chars. Husky lint-staged will run eslint --fix + prettier --write on staged `.ts` files.
