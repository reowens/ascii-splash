# v0.4.0 Roadmap — "From Engine to Canvas"

**Status**: 🚧 Phases 1 + 2 + 3 done on branch — Phases 4–9 planned
**Created**: May 9, 2026
**Last updated**: May 9, 2026 (Phase 3 marked complete)
**Target Release**: 6–9 weeks
**Focus**: Photo / video → ASCII rendering, multi-mode renderer, scene composition with photo backgrounds, modern terminal graphics protocols, shareability

---

## 🎯 Vision

Take ascii-splash from "a polished sibling of cmatrix/pipes.sh/asciiquarium" to **the terminal canvas that turns any image, video, or seed code into a shareable living scene**.

The leap is making **visual media a first-class input** alongside procedural patterns, then composing the two through the existing `SceneGraph`. No mainstream terminal app currently does that well. That's the moat.

We do this without rewriting the engine — the existing `Cell[][]` buffer, `Pattern` interface, and `SceneGraph` already match the architecture this needs (validated by reading `tarts/src/buffer.rs` and `tarts/src/common.rs`, which arrived at almost identical types independently).

## ✅ In Scope

- ✅ **`PhotoPattern`** — load any image and render it into the existing buffer
- ✅ **Five rendering modes** — ascii-ramp, half-block (default), braille, chafa-style symbol matcher, native graphics protocol pass-through
- ✅ **Edge-aware character selection** — implicit via 8×8 symbol matching (chafa technique)
- ✅ **Dithering** — Floyd-Steinberg (default) and Bayer ordered, used for 256-color and braille modes
- ✅ **Perceptual color matching** — CIELAB ΔE for palette quantization
- ✅ **Scene composition** — photo backgrounds layered with existing procedural patterns
- ✅ **Kitty graphics protocol** — detection + image emit (with iTerm2 + Sixel as opt-in)
- ✅ **Color-mask sprite format** — borrowed from asciiquarium, for richer hand-drawn scenes
- ✅ **Seeded PRNG plumbing** + share-code generator (`splash play AB7K2X9`)
- ✅ **asciinema `.cast` export** — frictionless sharing
- ✅ **GIF export** (deferred slot — only if effort allows)

## ❌ Out of Scope (deferred to v0.5+)

- ❌ Audio-reactive mode — defer to v0.5; cross-platform Node audio capture is genuinely painful and best done via `cava` subprocess once we have a real user signal asking for it
- ❌ SDF / raymarching scenes — at most one showpiece pattern (donut-style) in v0.4 if effort allows; full framework is v0.5
- ❌ Fluid simulation — single pattern at most; full interactive sandbox is v0.5
- ❌ Plugin / scripting system — premature without community demand. JSON-defined patterns yes, embedded JS/WASM no
- ❌ Video-to-ASCII — frame extraction via ffmpeg is straightforward but the UX (scrubbing, looping, audio sync) is its own project. Static images first.
- ❌ ML / Random Forest character selection — the chafa bitmap-matching algorithm is good enough and 12 lines instead of an ML pipeline
- ❌ Daily / time-of-day patterns — nice-to-have; trivial once seeded PRNG lands

## 🧠 Strategic Foundations (from code review)

We cloned six reference repos into `~/Development/packages/` and read the source. Three findings reshape the plan:

### Finding 1: Chafa's quality is one specific algorithm, ~300 LOC to port

`chafa/internal/chafa-symbol-renderer.c:98-113` is the entire quality story:

```c
// calc_cell_error_plain — chafa's whole quality story in 12 lines
for (i = 0; i < CHAFA_SYMBOL_N_PIXELS; i++) {
    guint8 p = *cov++;                    // 0 or 1 from symbol's 8×8 bitmap
    const ChafaPixel *p0 = block++;       // actual source pixel
    error += chafa_color_diff_fast(       // accumulate color error
        &color_pair->colors[p], &p0->col);
}
```

Each candidate Unicode symbol (`/`, `─`, `▘`, `█`, etc.) carries an **8×8 bitmap** stored in headers like `chafa-symbols-ascii.h:36-44`:

```c
{
    /* [!] */
    CHAFA_SYMBOL_TAG_ASCII | CHAFA_SYMBOL_TAG_LATIN, 0x21,
    CHAFA_SYMBOL_OUTLINE_8X8 (
        "        "
        "   X    "
        "   X    "
        "   X    "
        "   X    "
        "        "
        "   X    "
        "        ")
},
```

For each terminal cell, chafa pulls the corresponding 8×8 source-image patch, then for every candidate symbol computes `fg = mean(pixels where bitmap=1)`, `bg = mean(pixels where bitmap=0)`, and sums squared color distance. Lowest-error symbol wins. **This naturally selects `/` for diagonal edges, `─` for horizontal, `▘` for upper-left quadrants — without any explicit edge-detection pass.** The bitmap matching is implicit edge detection.

That's why chafa's output looks dramatically better than `jp2a` or `ascii-image-converter`. And it's portable.

### Finding 2: Half-block rendering is ~150 LOC, port verbatim from viuer

`viuer/src/printer/block.rs:36-151` — the entire algorithm:

```rust
const UPPER_HALF_BLOCK: &str = "\u{2580}";   // ▀
const LOWER_HALF_BLOCK: &str = "\u{2584}";   // ▄

// Even rows fill bg, odd rows fill fg, emit ▄
// Result: each terminal cell encodes 2 vertically stacked pixels
// (different colors via 24-bit fg + bg ANSI)
```

You take a `2W × H` source-image strip, walk pairs of rows, and emit `\x1b[38;2;r1;g1;b1m\x1b[48;2;r2;g2;b2m▄` per cell. **That's it.** Including transparency handling (cursor-skip on alpha=0) and the last-odd-row case (use `▀` instead), the whole printer is ~150 LOC. This is the right Day-1 default — gets us 2× vertical resolution over plain ASCII for free.

### Finding 3: Existing architecture is correct — no rewrite needed

`tarts/src/buffer.rs:1-107` is functionally identical to ascii-splash's:

```rust
pub struct Cell { pub symbol: char, pub color: Color, pub attr: Attribute }
pub struct Buffer { pub width: usize, pub height: usize, pub buffer: Vec<Cell> }
impl Buffer { pub fn diff(&self, other: &Buffer) -> Vec<(usize, usize, Cell)> {…} }

pub trait TerminalEffect {
    fn get_diff(&mut self) -> Vec<(usize, usize, Cell)>;
    fn update(&mut self);
    fn update_size(&mut self, width: u16, height: u16);
    fn reset(&mut self);
}
```

That's our `Pattern` interface in different syntax. Means **`PhotoPattern` slots in alongside `WavePattern` etc.**, and the existing dirty-rect plumbing is already there. `SceneGraph` already handles layering. No architectural change.

## 🏗️ Architecture

### `PhotoPattern` interface

```typescript
interface PhotoPatternConfig {
  source: string | Buffer; // file path or image bytes
  mode: 'ascii' | 'halfblock' | 'braille' | 'symbol' | 'auto';
  ramp?: string; // override char ramp (ascii mode)
  symbolTags?: SymbolTag[]; // restrict candidate set (symbol mode)
  edgeMode?: 'off' | 'overlay' | 'directional';
  dither?: 'none' | 'floyd-steinberg' | 'bayer-8' | 'bayer-16';
  invert?: boolean; // negative
  threshold?: number; // braille mode (0-255)
  colorBg?: boolean; // color the background instead of fg
  motion?: PhotoMotion; // zoom/pan/parallax (optional)
}

class PhotoPattern implements Pattern {
  name: string;
  render(buffer: Cell[][], time: number, size: Size, mousePos?: Point): void;
  reset(): void;
  getPresets(): PatternPreset[];
  applyPreset(presetId: number): boolean;
  // ...standard Pattern members
}
```

### Renderer dispatch (mode = `auto`)

```
1. Detect terminal capability once at startup → cached
2. If config.mode === 'auto':
     if Kitty supported && image is large       → kitty pass-through
     else if iTerm2 supported && macOS user     → iterm2 inline
     else                                       → halfblock (default)
3. If protocol pass-through:
     emit native protocol bytes; skip Cell[][] for that region
4. Else:
     decode → resize → optional dither/edge → fill Cell[][]
```

### Color pipeline

```
RGB pixel
  → optional dither (Floyd-Steinberg or Bayer)
  → quantize for output mode
       truecolor      → emit as-is
       256-color      → CIELAB ΔE nearest in 256-cube
       16-color       → ΔE nearest in ANSI 16
  → emit \x1b[38;2;r;g;bm  (truecolor)
       or \x1b[38;5;Nm     (256)
       or \x1b[3{N}m       (16)
```

### Scene composition

`PhotoPattern` becomes a `SceneLayer` (existing interface, `src/types/index.ts:254-260`) in the existing `SceneGraph`. Procedural patterns render _on top_:

```
Photo (beach.jpg)   zIndex=0  ────┐
                                  ├─→ SceneGraph (already z-orders + renders bottom-up)
WavePattern overlay  zIndex=10 ───┘
```

Composition rules (verified against `src/engine/SceneGraph.ts:104-114`):

- Photo layer: rendered to its assigned region with `zIndex=0` once on load, then frozen
- Overlay layer: renders normally each frame, **higher zIndex**
- **Transparency convention is space-character-only** — there is no alpha channel on `Cell`. SceneGraph renders in z-order with later layers overwriting earlier ones. The existing `SpriteManager.render` (`src/engine/SpriteManager.ts:138`) already follows this: `if (char === ' ') continue;` skips writes, leaving the layer below visible. **PhotoPattern + overlays follow the same rule.**
- Performance: photo layer pays its decode/render cost **once per resize**, not per frame

**Eligible overlay patterns** (sparse — naturally leave most cells blank):
Lightning, Fireworks, Rain, Snow, Starfield, DNA, Particle, Smoke, Snowfall, Quicksilver

**Not directly composable** (paint every cell — would obscure the photo):
Plasma, Wave, Tunnel, Spiral, Matrix, Maze, Life, LavaLamp, Metaball, scene-based patterns

**Plan for "dense" patterns**: add a `transparentBg: boolean` config option (Phase 3). When true, cells that would have rendered the pattern's _background_ color are left as `' '` instead. ~5-10 LOC per pattern. We pick 3-4 dense patterns to migrate in Phase 3 (Plasma + Matrix + Wave are the obvious wins for photo overlays); the rest can migrate post-v0.4 as users request them.

### Capability detection

Kitty/iTerm2/Sixel detection runs **once at startup**. Two-stage approach:

**Stage 1 — environment-variable fast paths** (zero-latency):

- `$KITTY_WINDOW_ID` → Kitty
- `$TERM_PROGRAM === 'iTerm.app'` → iTerm2 (but skip if `$TMUX` is also set — see below)
- `$TERM_PROGRAM === 'WezTerm'` → WezTerm (Kitty + iTerm2 capable)
- `$GHOSTTY_RESOURCES_DIR` → Ghostty (Kitty capable)
- `$VTE_VERSION` → VTE-based (gnome-terminal etc., no graphics protocol)

**Stage 2 — escape-sequence probe** if env vars are inconclusive:

- Send Kitty graphics query `\x1b_Gi=31,s=1,v=1,a=q,t=d,f=24;AAAA\x1b\\` followed by DA1 fallback `\x1b[c`
- **Timeout: 250 ms** (raised from initial 50 ms estimate — 50 ms produces false negatives over SSH and through tmux; `kitty icat` uses configurable longer waits)
- If only DA1 response arrives → no Kitty support → fall through to halfblock

**tmux handling**:

- If `$TMUX` is set, auto-run `tmux set-option -p allow-passthrough on` for the current pane (matching `kitty icat`'s behavior). Fail silently if no permission.
- **Skip iTerm2 inline-image protocol entirely when `$TMUX` is set** — this is a protocol-design limitation in iTerm2 (font metrics aren't multiplexer-visible), not a configuration issue, and it's never going to work
- Sixel inside tmux: works in tmux 3.5+ with allow-passthrough; treat as opt-in

Detection result is cached for the session — never repeated per-frame.

### Color-mask sprite format (asciiquarium technique)

For richer hand-drawn scenes (the existing `OceanBeach`, `Campfire`, `Aquarium` scenes), extend the existing `Sprite` interface (`src/types/index.ts:263-273`) with two **optional** fields:

```typescript
interface Sprite {
  // ...existing fields (position, velocity, frames, currentFrame, frameTime,
  // frameDuration, color, scale, active)

  // NEW (optional — backward compatible):
  colorMask?: string[][]; // parallel to frames, same dimensions
  palette?: Record<string, Color>; // { Y: yellow, W: white, c: cyan, ... }
}
```

`SpriteManager.render` (currently `src/engine/SpriteManager.ts:111-152`) gets one branch added:

```typescript
const cellColor = sprite.colorMask
  ? (sprite.palette?.[sprite.colorMask[currentFrame][lineIdx][charIdx]] ?? sprite.color)
  : sprite.color;
```

Existing sprites without `colorMask` keep the current single-color behavior unchanged. Net change: ~30 LOC in SpriteManager + ~5 LOC in `Sprite` type definition. No migration required for existing scenes.

`asciiquarium:1480-1486` is the reference for the substitution semantics.

### Seeded PRNG

Replace `Math.random()` calls in patterns with an injected PRNG (`mulberry32` or `xoshiro128**` — both <30 LOC). Pattern state at render becomes:

```typescript
{ patternId, presetId, themeId, seed: u32, configHash: u32 }
```

Encode that as ~10 chars base32 → share string. Decode → restore exact same scene.

### Phase 4: Chafa-style symbol matcher (Weeks 4–5)

**Goal**: Wow-mode rendering. Implicit edge detection. The screenshot-bait mode.

**Tasks**:

- [ ] Hand-author 30–50 8×8 bitmap symbols (ASCII subset + half-blocks + quadrants + key Unicode block characters). Reference Terminus glyphs for shape, but author from scratch (chafa is LGPL, we are MIT)
- [ ] Implement `SymbolMap` data structure with tag-based filtering (`BLOCK | LATIN | ASCII | BRAILLE`)
- [ ] Implement `SymbolEvaluator` (port chafa-symbol-renderer.c:98-268, plain version, ~200 LOC)
- [ ] Add `mode: 'symbol'` to `PhotoPattern`
- [ ] User control: which tag set to use (`--symbol-set ascii`, `--symbol-set block`, etc.)
- [ ] Six presets across symbol-set / contrast combinations
- [ ] Performance test: 80×24 cells × 50 candidate symbols at 30fps = 96k error evaluations/s
- [ ] Tests: symbol selection correctness on known patches (vertical stripe → `|`, diagonal → `/`)

**Perf reality (already measured)**: A pessimistic plain-JS microbenchmark (Node 25, no SIMD, no early-exit pruning, two passes per cell) achieves **49 fps at 80×24 with 50 symbols** = ~20.4 ms/frame. Throughput ≈ 4.7M symbol-evals/sec.

| Terminal size | Symbol evals / frame @ 30fps | Budget verdict                                                   |
| ------------- | ---------------------------- | ---------------------------------------------------------------- |
| 80 × 24       | 96k                          | ✅ comfortable (5×+ headroom)                                    |
| 100 × 30      | 150k                         | ✅ ok                                                            |
| 120 × 40      | 240k                         | ⚠️ tight — needs early-exit pruning, sub-sampling, or 15fps mode |
| 160 × 50      | 400k                         | ❌ requires fast-path (top-N candidate screen) or drop to 15fps  |

**Implication**: ship Phase 4 at 80×24 / 100×30 first. If users on big terminals complain, add a fast-path that pre-screens candidates by mean-brightness before doing full 64-pixel error eval (chafa's actual production technique, gated behind `N_CANDIDATES_MAX = 8`).

**Deliverable**: A "premium" rendering mode that produces output competitive with chafa. The mode that makes screenshots viral.

## 🧮 Algorithm Appendix

### A. Half-block renderer (default photo mode)

Source: `viuer/src/printer/block.rs:36-151`.

```
For each pair of rows (top, bottom) in source image (resized to terminal width × 2):
  For each column:
    if both pixels opaque:
      emit "\x1b[38;2;{rt};{gt};{bt}m\x1b[48;2;{rb};{gb};{bb}m▄"
    elif only top opaque:
      emit "\x1b[38;2;{rt};{gt};{bt}m▀"
    elif only bottom opaque:
      emit "\x1b[38;2;{rb};{gb};{bb}m▄"
    else:
      emit "\x1b[1C"      // skip transparent
  emit "\r\n"
```

Cost: O(W × H), one ANSI sequence per cell. Output is one terminal row per two source rows.

### B. Braille bit-pack (8× resolution mode)

Source: `drawille/drawille.py:43-129`, `ascii-image-converter/.../ascii_conversions.go:264-283`.

```
pixel_map = [
  [0x01, 0x08],
  [0x02, 0x10],
  [0x04, 0x20],
  [0x40, 0x80]
]
braille_offset = 0x2800

# To set pixel (x, y) globally:
cell_col = x // 2
cell_row = y // 4
cell[cell_row][cell_col] |= pixel_map[y % 4][x % 2]

# To render cell:
char = chr(0x2800 + cell[cell_row][cell_col])
```

Each terminal cell encodes 8 dots (2 wide × 4 tall). For a photo: threshold pixels at some brightness level (or use Floyd-Steinberg first → 1-bit), then bit-pack into braille chars. Color comes from the **average color of the lit dots** in each cell.

### C. Floyd-Steinberg dithering

```
For each pixel (x, y) in row-major order:
  oldP = pixel[x][y]
  newP = quantize(oldP)         // nearest in target palette
  pixel[x][y] = newP
  err = oldP - newP
  pixel[x+1][y  ] += err * 7/16
  pixel[x-1][y+1] += err * 3/16
  pixel[x  ][y+1] += err * 5/16
  pixel[x+1][y+1] += err * 1/16
```

Preserves average brightness exactly; produces "worm" artifacts in flat regions but excels on photographs.

### D. Bayer ordered dithering

`chafa/internal/chafa-dither.c:54-82`. Pre-generate an N×N matrix (N=8 or 16) where entry (i,j) is the index in a recursive Bayer pattern divided by N². At render:

```
threshold_mod = bayer[y % N][x % N] * intensity   // signed [-128, +127]
pixel.r = clamp(pixel.r + threshold_mod, 0, 255)
pixel.g = clamp(pixel.g + threshold_mod, 0, 255)
pixel.b = clamp(pixel.b + threshold_mod, 0, 255)
// then quantize as normal
```

Produces structured patterns ideal for retro / 1-bit aesthetics. Chafa applies the _same_ threshold to all three channels (preserves hue); a per-channel variant would shift hues more.

### E. Chafa symbol-matcher (the wow-mode core)

Source: `chafa/internal/chafa-symbol-renderer.c:98-268`.

```
Symbol = { tags: bitmask, codepoint: char, bitmap: u8[64] }   // 8×8

// Per terminal cell:
patch = source_image.crop(cell_x * 8, cell_y * 8, 8, 8)
best_error = INF
best_sym = ' '

for sym in symbol_map.filter_by_tags(active_tags):
  // 1. Compute fg / bg means weighted by bitmap
  fg = mean(patch[i] for i in 0..63 if sym.bitmap[i] == 1)
  bg = mean(patch[i] for i in 0..63 if sym.bitmap[i] == 0)

  // 2. Sum squared color error
  error = 0
  for i in 0..63:
    expected = fg if sym.bitmap[i] else bg
    error += color_diff_sq(expected, patch[i])

  if error < best_error:
    best_error = error
    best_sym = sym
    best_colors = (fg, bg)

emit ansi(best_colors.fg, best_colors.bg, best_sym.codepoint)
```

`color_diff_sq` is squared Euclidean in linear RGB (or CIELAB for higher quality). Chafa's optimization: only consider top-N candidates after a fast first-pass screen — but for ~50 symbols on 80×24 cells we don't need it.

### F. Kitty graphics protocol handshake

Source: `viuer/src/printer/kitty.rs:140-263`.

```
// Send query + DA1 fallback in one batch:
write: "\x1b_Gi=31,s=1,v=1,a=q,t=d,f=24;AAAA\x1b\\"   // 1×1 RGB image query
write: "\x1b[c"                                        // primary device attrs

// Read keys until either:
//   - "_Gi=31;OK\\"  → Kitty supported (graphics)
//   - "[?...c"       → only DA1 came back, no Kitty

// If supported, optionally probe local-file path support:
// (write a 1-byte tempfile, query with t=t)

// Local support: pass image as file path (small payload, fast)
write: "\x1b_Gf=32,s={W},v={H},c={cellW},r={cellH},a=T,t=t;{base64(filepath)}\x1b\\"

// Remote support: chunk image data 4096 bytes at a time
write: "\x1b_Gf=32,a=T,t=d,s={W},v={H},c={cellW},r={cellH},m=1;{first_chunk}\x1b\\"
write: "\x1b_Gm=1;{next_chunk}\x1b\\"  // m=1 means "more chunks"
write: "\x1b_Gm=0;{last_chunk}\x1b\\"  // m=0 means "this is the last"
```

Detection runs **once at startup** with a ~50ms timeout. Cached for the session.

### G. Color-mask sprite format

Source: `asciiquarium:1480-1486`.

```
// Shape — ASCII art:
"  /\__/\  "
" ( o.o )  "
"  > ^ <   "

// Mask — same dimensions, characters are color slots:
"  YYWWWY  "
" WWcCcWW  "
"  W W W   "

// Palette (themable):
{ Y: yellow, W: white, c: cyan, C: deepCyan }

// Render: for each (x, y), emit shape[y][x] colored by palette[mask[y][x]]
```

Mask characters that are not in the palette default to the active theme's primary color. Empty space in mask = use background.

### ✅ `PlasmaPattern` char-bin off-by-one — resolved on `feature/v0.4.0-phase1-photo-pattern`

`PlasmaPattern.ts` previously used `floor(intensity * (length - 1))`, which
made the `' '` bin reachable only at `intensity === 1.0` (measure zero). Fixed
to `Math.min(length - 1, Math.floor(intensity * length))` so all 8 chars now
get uniform 1/8-wide bins.

Approach: option (1) — fix off-by-one + revert workaround. `transparentBg` is
now the clean `if (char === ' ') continue;` predicate, with standalone Plasma
gaining ~12.5% blank cells (rendered as themed background space). All 12
presets affected uniformly. Existing tests (2216) all green; the Phase 3
`transparentBg` tests now exercise the real ' ' bin instead of incidentally
catching '·'.
