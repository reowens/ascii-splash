# v0.2.0 ESM Migration Plan

**Status**: Planning Phase - REFINED WITH 2025 BEST PRACTICES  
**Target Release**: TBD (v0.1.5 completed November 4, 2025)  
**Type**: MAJOR VERSION (Breaking Change)  
**Priority**: HIGH (enables modern dependency updates)  
**Last Updated**: November 4, 2025 (Refined with Perplexity research)

## Overview

Migrate ascii-splash from CommonJS to ESM (ECMAScript Modules) to enable modern dependency updates, particularly conf v15+, and align with the JavaScript ecosystem's future direction.

## 2025 ESM Best Practices Research

**Research Date**: November 4, 2025  
**Source**: Perplexity AI with verified 2025 sources

### Key Findings

1. **Pure ESM Recommended**: For CLI apps in 2025, pure ESM is the standard. Dual CJS/ESM adds complexity with minimal benefit.

2. **Node.js 20.11+ Native Features**:
   - `import.meta.dirname` and `import.meta.filename` now native (no need for `fileURLToPath`)
   - Top-level await fully supported
   - ESM performance parity with CommonJS achieved

3. **Jest with tsx > ts-jest**: 
   - `tsx` provides faster, zero-config TypeScript + ESM support
   - `ts-jest` ESM support can be brittle in complex projects
   - Industry trend moving toward tsx for ESM projects

4. **ModuleResolution Confirmed**:
   - Use `"node16"` for Node.js applications
   - Only use `"bundler"` for Vite/Webpack/Bun contexts

5. **Dependencies Verified ESM-Compatible**:
   - ✅ terminal-kit 3.1.2: Supports ESM imports
   - ✅ chalk 5.3.0: Pure ESM only (since v5)
   - ✅ commander 14.0.2: Full ESM support

6. **Performance**: ESM vs CommonJS performance is equivalent in 2025 for CLI apps

7. **conf v10 → v15 Breaking Changes**:
   - Primary: ESM-only (no `require()`)
   - Secondary: Possible config location changes (verify in testing)
   - API adjustments: Review migration guide during implementation

## Motivation

### Primary Driver: Dependency Updates
- **conf v10.2.0 → v15.0.2** requires ESM (current blocker)
- conf v15+ is ESM-only (`"type": "module"`)
- Cannot update conf without full ESM migration

### Secondary Benefits
- ESM is the future of Node.js modules (officially recommended)
- Better tree-shaking and bundle optimization
- Improved static analysis capabilities
- Native browser compatibility (potential future feature)
- Aligns with modern tooling and ecosystem direction
- Enables future adoption of other modern ESM-only packages

### __dirname Replacement

**⚠️ UPDATED 2025**: Node.js 20.11+ provides native `import.meta.dirname` and `import.meta.filename`

**Before (CommonJS):**
```typescript
import * as path from 'path';
const configPath = path.join(__dirname, '..', 'config', 'defaults.json');
```

**After (ESM) - Node.js 20.11+ Native (Recommended):**
```typescript
import * as path from 'path';

// Native Node.js 20.11+ properties - no imports needed!
const configPath = path.join(import.meta.dirname, '..', 'config', 'defaults.json');
const currentFile = import.meta.filename;
```

**Alternative (Legacy compatibility, if needed for Node.js <20.11):**
```typescript
import * as path from 'path';
import { fileURLToPath } from 'node:url';

const __filename = fileURLToPath(import.meta.url);
const __dirname = fileURLToPath(new URL('.', import.meta.url));
const configPath = path.join(__dirname, '..', 'config', 'defaults.json');
```

**Note**: Since our project requires Node.js 20+, use the native `import.meta.dirname` approach.

### Jest Configuration

**⚠️ UPDATED 2025**: Use `tsx` for cleaner, faster ESM support instead of ts-jest

**Before (jest.config.js):**
```javascript
module.exports = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  roots: ['<rootDir>/src', '<rootDir>/tests'],
  testMatch: ['**/?(*.)+(spec|test).ts'],
  // ...
};
```

**After (jest.config.mjs) - Using tsx (Recommended for 2025):**
```javascript
export default {
  testEnvironment: 'node',
  extensionsToTreatAsEsm: ['.ts'],
  transform: {
    '^.+\\.tsx?$': 'tsx'  // Zero-config TypeScript + ESM support
  },
  moduleNameMapper: {
    '^(\\.{1,2}/.*)\\.js$': '$1',  // Map .js imports to .ts files
  },
  roots: ['<rootDir>/src', '<rootDir>/tests'],
  testMatch: ['**/?(*.)+(spec|test).ts'],
  collectCoverageFrom: [
    'src/**/*.ts',
    '!src/**/*.d.ts',
    '!src/types/**'
  ],
  coverageThreshold: {
    global: {
      branches: 80,
      functions: 80,
      lines: 80,
      statements: 80
    }
  }
};
```

**Dependencies Update:**
```bash
npm install --save-dev tsx  # Add tsx for Jest
# Keep ts-jest if needed for other tools, or remove it
```

**Why tsx over ts-jest?**
- Faster test execution
- Zero-config ESM support
- Better handling of TypeScript + ESM edge cases
- Recommended by 2025 ESM migration guides

## 2025 ESM Migration Best Practices Summary

Based on research and industry trends, here are the key recommendations:

### ✅ DO (2025 Recommendations)
1. **Use tsx for Jest** - Faster, cleaner than ts-jest for ESM
2. **Use native `import.meta.dirname`** - Node.js 20.11+ feature, no imports needed
3. **Use `"moduleResolution": "node16"`** - Correct for Node.js CLI apps
4. **Go pure ESM** - Don't maintain dual CJS/ESM for CLI apps
5. **Remove "use strict"** - Redundant in ESM (enforced by default)
6. **Add `.js` extensions** - Required for ESM imports, even in TypeScript
7. **Verify dependencies first** - Ensure all deps support ESM before starting

### ❌ DON'T (Common Pitfalls)
1. **Don't use ts-jest for ESM** - More complex, slower than tsx
2. **Don't use fileURLToPath** - Use native `import.meta.dirname` instead (Node.js 20+)
3. **Don't use "bundler" resolution** - Only for Vite/Webpack, not Node.js
4. **Don't maintain dual CJS/ESM** - Adds complexity, minimal benefit for CLI
5. **Don't forget `.js` extensions** - ESM requires explicit file extensions
6. **Don't mix CJS and ESM** - Choose one, stick to it

### Performance Notes
- ESM vs CommonJS performance is equivalent in 2025 for CLI apps
- No meaningful cold start or import speed differences
- Top-level await is now fully optimized
